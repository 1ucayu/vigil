# VigilMarket — Fidelity App

A small native Android app used by the [Vigil](../README.md) project as a
controlled benchmark target. Vigil's offline pipeline (APK static priors,
runtime UI XML, screenshots, action traces, replay) is exercised against this
app to test whether it can faithfully construct and validate an FSM
`M_A = <S, s0, Sigma, delta, Gamma, I, rho>`.

The app is intentionally native and deterministic, but it now uses a modern
Material3 shell: top app bar, bottom navigation, cards, and summary panels.
It does **not** talk to the network or any database, has no authentication,
and ships only seeded in-memory data.

## Why a separate Android project (not Python)

Vigil's pipeline relies on real Android UIAutomator XML, accessibility nodes,
and APK metadata. Cross-platform UI frameworks would not produce the same
structure, so this app is kept outside `src/vigil/` as its own Kotlin +
Jetpack Compose project. It is intentionally not wired into the Python
pipeline yet.

## Build / install / launch

```bash
cd fidelity_app
./gradlew assembleDebug
./gradlew installDebug
adb -s emulator-5554 shell monkey -p edu.hku.vigil.fidelity -c android.intent.category.LAUNCHER 1
```

### Deterministic clean-launch flow (recommended between replay runs)

```bash
adb -s emulator-5554 shell am start -S -n edu.hku.vigil.fidelity/.MainActivity
```

Or, in-app: open `Settings → Reset demo` (`testTag = settings.reset_demo`),
which resets the cart, selected product, quantity, address, dialog state,
search query, and **also** resets the inert toggles `high_contrast` and
`confirm_payments` to `false`. A fresh launch and an in-app reset produce the
same UI fingerprint.

## Implemented states

| State              | Screen marker             | Notes                                |
|--------------------|---------------------------|--------------------------------------|
| `home`             | `screen:home`             | Entry. Buttons to all sub-screens.   |
| `search`           | `screen:search`           | Local `contains` filter.             |
| `catalog`          | `screen:catalog`          | 5 seeded products.                   |
| `product_detail`   | `screen:product_detail`   | Template state, parameter `product_id`. |
| `cart_empty`       | `screen:cart_empty`       | No item selected.                    |
| `cart`             | `screen:cart`             | Qty +/- (clamped 1..9), remove, checkout. |
| `address_select`   | `screen:address_select`   | 3 seeded addresses.                  |
| `payment_confirm`  | `screen:payment_confirm`  | Shows product, qty, address, total.  |
| `payment_success`  | `screen:payment_success`  | Terminal.                            |
| `orders`           | `screen:orders`           | 2 seeded past orders.                |
| `settings`         | `screen:settings`         | Inert toggles + `reset_demo`.        |
| `remove_dialog`    | dialog over `cart`        | Confirmation for remove.             |
| `payment_dialog`   | dialog over `payment_confirm` | Confirmation for pay.            |

The app shell includes a Material3 bottom navigation bar on every non-dialog
state. Its global actions are tagged `nav.open_home`, `nav.open_search`,
`nav.open_catalog`, `nav.open_cart`, `nav.open_orders`, and
`nav.open_settings`; these are documented in `gold/fsm.json` under
`global_navigation`.

System back is implemented via Compose `BackHandler` and is deterministic;
the back targets are first-class transitions of kind `system_back` in
`gold/fsm.json`. From `home`, system back exits the app; in any open dialog
it dismisses the dialog before any screen transition.

## Canonical actions (selected)

```
click(home.open_catalog)
click(catalog.product_row.espresso.open)
click(product_detail.add_to_cart)
click(cart.checkout)
click(address_select.address.home)
click(payment_confirm.pay)
click(payment_dialog.confirm)
click(cart.remove_item)
click(remove_dialog.confirm)
click(settings.reset_demo)
click(nav.open_catalog)
input(search.query, "tea")
system_back(system.back)
```

Every navigable button carries a stable `Modifier.testTag(...)`. The root
`Surface` enables `semantics { testTagsAsResourceId = true }`, so test tags
appear as `resource-id` in UIAutomator dumps. Each screen also emits a
visible-and-readable `screen:<id>` text node (`testTag = screen_marker`).

## Stable identifiers / fidelity notes

- All seed data (`Catalog`, `Addresses`, `Orders`) is in-memory and
  deterministic — no random IDs, no timestamps.
- Money is integer cents internally; displayed as `$X.YZ`. Total = unit price
  × quantity (no taxes/fees), so guard arithmetic is exact.
- Dialogs use Material3 `AlertDialog` with confirm/dismiss buttons that carry
  their own `testTag`s (`<dialog>.confirm`, `<dialog>.cancel`).
- The two Settings switches (`settings.high_contrast`,
  `settings.confirm_payments`) are flagged `inert: true` in `gold/fsm.json`
  and **do not** affect navigation, dialogs, payments, guards, or transition
  topology in v1.

## Gold artifacts

`gold/` holds evaluator-only ground truth: see [gold/README.md](gold/README.md).
The app does **not** read these files at runtime.

## Scope

No network, no database, no auth, no external images, no runtime permissions,
no integration with the existing Vigil Python pipeline yet. Future work can
ingest this app's APK and traces, but no Python files in `src/vigil/` were
modified to introduce it.
