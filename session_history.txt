‚Źļ Now the picture is clear. Here are the two root causes:

  Root Cause 1: current_screen_id tracking is wrong. After executing an action
  that doesn't change screens (same fingerprint), current_screen_id gets set to
  a new screen ID (e.g., scr_0003), but the frontier has actions queued for
  scr_0001. So the explorer tries to navigate to scr_0001 even though it's
  already on that screen. The main Settings screen was captured 62 times because
   of this.

  Root Cause 2: Navigation can only go backward, not forward. When the frontier
  has actions for a deeper screen (e.g., Wi-Fi settings), the only strategy is
  pressing back ‚ÄĒ which moves AWAY from the target. The explorer backs all the
  way to the home screen (captured 27 times), restarts the app, lands on the
  main screen, and fails because that's not the target.

  The fix needs:
  1. Fingerprint-to-screen-id mapping ‚ÄĒ correctly identify where we are
  regardless of which capture
  2. Replay-based forward navigation ‚ÄĒ record how each screen was reached,
  replay that path
  3. Current-screen priority ‚ÄĒ exhaust actions from current screen before
  navigating elsewhere
